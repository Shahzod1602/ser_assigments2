## SLIDE 3 — Introduction: Why Process Models Matter
**Title:** Why Do Process Models Matter?

- Software projects frequently fail due to poor planning and unclear processes
- A process model provides structure, guidance, and repeatability
- The right model reduces risk, saves cost, and improves quality
- Different projects need different approaches

**Key Fact:** The Standish Group CHAOS Report consistently shows that over 60% of software projects experience delays, budget overruns, or outright failure.

## SLIDE 5 — Waterfall Model: Overview
**Title:** The Waterfall Model

- Introduced by Winston Royce (1970)
- Linear, sequential process — each phase completed before the next starts
- Named for the way progress flows downward like a waterfall

**The 6 Phases:**
1. Requirements → 2. System Design → 3. Implementation → 4. Testing → 5. Deployment → 6. Maintenance

*Design tip: Use a downward-flowing arrow diagram showing all 6 phases.*

---

## SLIDE 6 — Waterfall: Characteristics & Fit
**Title:** Waterfall — Key Characteristics

**Advantages:**
- Simple and easy to manage
- Strong documentation at every phase
- Clear milestones and deliverables
- Works well with large, fixed-scope contracts

**Disadvantages:**
- Inflexible — changes are very costly
- Customer sees product only at the end
- High risk if requirements are wrong
- Testing is done too late

**Best For:** Fixed requirements, government contracts, payroll systems, short projects

*Design tip: Two-column layout — green checkmarks for advantages, red X for disadvantages.*

## SLIDE 7 — Spiral Model: Overview
**Title:** The Spiral Model

- Developed by Barry Boehm (1986)
- Combines iterative development with systematic risk management
- Each "loop" of the spiral is a complete development cycle
- Focuses on identifying and resolving risks early

**4 Quadrants per Spiral:**
1. Planning → 2. Risk Analysis → 3. Engineering → 4. Evaluation

*Design tip: Use an actual spiral diagram with the 4 quadrants labeled.*

---

## SLIDE 8 — Spiral Model: Characteristics & Fit
**Title:** Spiral — Key Characteristics

**Advantages:**
- Excellent risk management at every cycle
- Flexible — new requirements can be added in subsequent spirals
- Prototyping reduces chance of building the wrong system
- Strong customer involvement per cycle

**Disadvantages:**
- Very expensive and resource-intensive
- Requires specialized risk analysis skills
- Complex to manage
- Not suitable for small projects

**Best For:** Large-scale, high-risk systems — defense, aerospace, banking infrastructure

*Design tip: Spiral visual in the background with bullet points overlaid.*

## SLIDE 9 — Agile (Scrum): Overview
**Title:** Agile Model — Scrum Framework

- Agile Manifesto published in 2001
- Scrum is the most widely used Agile framework
- Work is organized into short iterations called **Sprints** (1–4 weeks)
- Each sprint delivers a working, usable increment of the product

**The Scrum Cycle:**
Backlog → Sprint Planning → Sprint (1–4 weeks) → Review → Retrospective → Repeat

**3 Key Roles:** Product Owner | Scrum Master | Development Team

*Design tip: Use a circular sprint cycle diagram.*

---

## SLIDE 10 — Agile (Scrum): Characteristics & Fit
**Title:** Scrum — Key Characteristics

**Advantages:**
- Highly flexible — changes welcome at any point
- Working software delivered every sprint
- Continuous customer feedback
- High team collaboration
- Faster time to market for core features

**Disadvantages:**
- Hard to predict final cost upfront
- Requires constant customer involvement
- Risk of scope creep
- Minimal documentation can cause maintenance issues
- Difficult to scale without additional frameworks

**Best For:** Startups, mobile apps, web products, evolving requirements

*Design tip: Use agile/kanban board imagery in the background.*

## SLIDE 13 — Banking System → Waterfall (+Spiral elements)
**Title:** Banking System → Waterfall Model

**Why?**
- Requirements are defined upfront by legal/regulatory teams
- Strict compliance needs heavy documentation (PCI-DSS, regulations)
- Changes to a live banking system are highly controlled
- Auditors need clear records of every phase
- Spiral elements help manage risk in complex modules

**Real Example:** Core banking system upgrades at HSBC or JPMorgan Chase use structured, plan-driven processes with rigorous documentation and sign-offs.

*Design tip: Bank/finance imagery. Use green color theme for this slide.*

---

## SLIDE 14 — Mobile Startup App → Agile (Scrum)
**Title:** Mobile Startup App → Agile (Scrum)

**Why?**
- Requirements evolve rapidly based on user feedback
- Need to ship an MVP quickly to validate the idea
- Small, co-located team — ideal Scrum environment
- Investors want to see frequent, tangible progress
- The market changes fast — Agile lets you pivot

**Real Example:** Instagram initially launched with only photo sharing — new features (Stories, Reels) were added iteratively based on user behavior and competitive pressure.

*Design tip: Smartphone/app imagery. Use blue/purple color theme.*

---

## SLIDE 15 — Government Defense System → Spiral
**Title:** Government Defense System → Spiral Model

**Why?**
- Extremely high risk — failures can cost lives
- Requirements are complex and not fully known at the start
- Prototyping allows evaluation before full commitment
- Multi-year projects with large budgets
- Aligns with defense procurement standards (MIL-STD, DO-178C)

**Real Example:** The development of the F-35 fighter jet's software (14+ million lines of code) used an iterative, risk-managed approach across multiple development cycles spanning decades.

*Design tip: Defense/military imagery. Use dark grey/green color theme.*

## SLIDE 17 — Role 1: Project Manager
**Title:** Project Manager

**Responsibilities:**
- Plans scope, timeline, and budget
- Manages risks and resources
- Primary contact between team and stakeholders
- Tracks progress and ensures quality

**Skills:** Leadership, communication, risk management, Jira/MS Project

**Contribution:** Keeps the team on track and aligned with business goals

**Real Example:** Microsoft PMs coordinate across hundreds of engineers to deliver Windows updates on schedule, managing cross-team dependencies and blockers.

*Design tip: Use a "compass" or "roadmap" icon.*

---

## SLIDE 18 — Role 2: Software Architect
**Title:** Software Architect

**Responsibilities:**
- Designs overall system structure and architecture
- Makes high-level technology decisions
- Sets coding standards and architectural guidelines
- Reviews code for architectural conformance

**Skills:** Design patterns, system integration, API design, multiple languages

**Contribution:** Prevents costly structural mistakes — defines how the system grows

**Real Example:** Netflix architects designed the microservices architecture that allows 200M+ users to stream simultaneously with high reliability.

*Design tip: Use a "blueprint" or "building" icon.*

---

## SLIDE 19 — Role 3: Developer
**Title:** Developer (Software Engineer)

**Responsibilities:**
- Writes, tests, and maintains application code
- Implements features from specifications/user stories
- Participates in code reviews
- Debugs and resolves defects

**Skills:** Programming languages (Python, Java, JS), Git, algorithms, design patterns

**Contribution:** Converts requirements and designs into a working product

**Real Example:** Google engineers use mandatory peer code review (via Gerrit) before any code merges — enabling billion-user scale products with minimal downtime.

*Design tip: Use a "code/laptop" icon.*

---

## SLIDE 20 — Role 4: QA Engineer
**Title:** QA Engineer

**Responsibilities:**
- Designs and executes test plans and test cases
- Performs manual and automated testing
- Documents and tracks defects to resolution
- Ensures non-functional requirements are met (performance, security)

**Skills:** Selenium, JUnit, Postman, JMeter, Jira, scripting

**Contribution:** Acts as the safety net — catches defects before they reach users

**Real Example:** Amazon uses chaos engineering (deliberately injecting failures) to verify platform stability, supporting thousands of deployments per day with minimal outages.

*Design tip: Use a "magnifying glass" or "shield" icon.*

---

## SLIDE 21 — Role 5: Business Analyst
**Title:** Business Analyst (BA)

**Responsibilities:**
- Elicits and documents business requirements
- Translates business needs into technical specifications
- Creates user stories, use cases, and process flows
- Bridges the gap between business and technical teams

**Skills:** Communication, UML/BPMN modeling, requirements elicitation, analytical thinking

**Contribution:** Ensures the team builds the right product, not just a correct one

**Real Example:** At JPMorgan Chase, BAs translate complex regulatory changes (e.g., new reporting rules) into functional specs that engineering teams use to update core banking systems.

*Design tip: Use a "bridge" or "conversation" icon.*

---

## SLIDE 22 — Role 6: DevOps Engineer
**Title:** DevOps Engineer

**Responsibilities:**
- Builds and maintains CI/CD pipelines
- Manages cloud infrastructure and containerization
- Monitors system performance in production
- Automates build, test, and deployment processes

**Skills:** Jenkins/GitHub Actions, AWS/Azure, Docker, Kubernetes, Terraform, Prometheus

**Contribution:** Reduces time from code to deployment; ensures reliability at every release

**Real Example:** Spotify's DevOps infrastructure allows 300+ microservices to be independently deployed multiple times per day by autonomous Scrum teams — making the "Spotify model" possible.

*Design tip: Use an "infinity loop" or "pipeline" icon (DevOps symbol).*

## SLIDE 23 — How Roles Interact with Process Models
**Title:** Roles × Process Models

| Role | Waterfall | Spiral | Agile (Scrum) |
|---|---|---|---|
| Project Manager | Plans entire project upfront | Manages each spiral cycle | Scrum Master facilitates |
| Software Architect | Designs full system before coding | Refines architecture each spiral | Guides architecture per sprint |
| Developer | Implements entire design at once | Builds incrementally per spiral | Delivers features per sprint |
| QA Engineer | Tests at the end | Tests at each spiral evaluation | Tests within every sprint |
| Business Analyst | Gathers all requirements upfront | Reviews and refines per spiral | Writes and refines user stories |
| DevOps Engineer | Manages single deployment | Supports multiple prototypes | Automates continuous deployment |

*Design tip: Color-coded table showing how the same roles behave differently per model.*

## SLIDE 24 — Key Takeaways
**Title:** Key Takeaways

**On Process Models:**
- No single model is universally best — context determines the right choice
- Waterfall: structure and documentation; Spiral: risk and complexity; Agile: speed and flexibility

**On Team Roles:**
- Every role is essential — removing one creates blind spots
- The right people using the right process = project success

**The Fundamental Truth:**
> "Software is built by people. Process models are only as good as the team that follows them."
